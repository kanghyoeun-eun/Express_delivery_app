# AGENTS.md

## Project Overview

- 이 저장소는 배달특급 모바일 하이파이 프로토타입이다.
- 정적 웹 프로젝트이며 주요 파일은 `index.html`, `styles.css`, `script.js`다.
- 현재 Production Branch는 `main`을 기준으로 한다.

## General Rules

- 작업 시작 전 저장소 구조, 현재 브랜치, Git 변경사항을 확인한다.
- 기존 코드와 디자인 패턴을 우선하여 따른다.
- 요청 범위를 벗어난 수정이나 리팩터링을 임의로 수행하지 않는다.
- 사용자가 수정 금지 또는 로컬 작업만을 요청하면 커밋, 푸시, 배포하지 않는다.
- 사용자 또는 다른 작업자가 만든 기존 변경사항을 삭제하거나 되돌리지 않는다.

## GitHub Issue Workflow

- 작업 시작 전에 현재 저장소의 열린 GitHub Issues를 확인한다.
- 사용자가 특정 이슈를 지정하면 해당 이슈를 최우선 작업 기준으로 사용한다.
- 지정된 이슈가 없다면 현재 요청과 직접 관련된 이슈만 참고한다.
- 이슈 내용과 사용자 요청이 충돌하면 사용자 요청을 우선하되 충돌 내용을 보고한다.
- 이슈에 없는 추가 작업을 임의로 수행하지 않는다.
- 작업과 관련된 이슈가 있다면 이슈 번호를 브랜치, 커밋 또는 PR 설명에 연결한다.
- 작업 완료 후 관련 이슈에 변경사항, 검증 결과, PR URL, Vercel 배포 URL을 댓글로 남긴다.
- 완료 조건을 모두 충족한 경우에만 관련 이슈를 닫는다.
- GitHub 접근 권한이나 인증이 없으면 임의로 우회하지 말고 사용자에게 보고한다.

## Implementation Workflow

1. 현재 브랜치와 Git 변경사항을 확인한다.
2. 관련 GitHub Issues를 확인한다.
3. 별도 작업 브랜치를 생성한다.
4. 요청된 변경사항만 구현한다.
5. 로컬에서 기능, 레이아웃, 이미지 로딩을 검증한다.
6. 변경사항을 커밋하고 GitHub에 푸시한다.
7. Pull Request를 생성한다.
8. PR의 변경사항과 검증 결과를 확인한다.
9. `main` 반영 후 Vercel Production 배포를 확인한다.
10. 실제 배포 URL에서 변경사항을 최종 검증한다.

## Git Rules

- `main` 브랜치에 직접 커밋하거나 강제 푸시하지 않는다.
- 작업마다 별도 브랜치를 사용한다.
- 관계없는 변경사항을 하나의 커밋에 포함하지 않는다.
- 커밋 메시지는 변경 목적을 명확하게 설명한다.
- 테스트 또는 검증에 실패한 변경사항을 푸시하거나 병합하지 않는다.
- 사용자의 명시적 승인 없이 기존 커밋 기록을 변경하지 않는다.

## Vercel Deployment

- GitHub 저장소와 연결된 Vercel 프로젝트를 사용한다.
- 작업 브랜치 푸시 후 Preview 배포 상태와 URL을 확인한다.
- PR 병합 후 Production 배포 상태를 확인한다.
- Production 배포가 성공한 경우 실제 URL에서 주요 기능과 이미지 로딩을 검증한다.
- 배포 실패 시 작업을 완료 처리하지 말고 실패 원인과 로그를 보고한다.
- GitHub 또는 Vercel 권한이 없으면 필요한 권한과 미완료 작업을 사용자에게 보고한다.

## Verification

- 수정한 기능의 주요 사용자 흐름을 직접 검증한다.
- 브라우저 콘솔 오류와 이미지 로딩 실패 여부를 확인한다.
- 기존 주요 화면과 상호작용에 회귀가 없는지 확인한다.
- 검증하지 못한 항목은 완료된 것처럼 보고하지 않는다.

## Completion Report

작업 완료 시 다음 항목을 보고한다.

- 변경사항 요약
- 관련 GitHub Issue
- 작업 브랜치와 커밋
- Pull Request URL
- 수행한 검증과 결과
- Vercel Preview 및 Production 배포 상태
- 실제 배포 URL
- 남아 있는 문제 또는 미검증 항목
